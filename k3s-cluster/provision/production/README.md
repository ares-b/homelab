# k3s-cluster

OpenTofu/Terraform that clones the Packer `ubuntu-k3s` template into a small k3s
cluster on Proxmox and bootstraps k3s through cloud-init.

Topology: one server (control-plane, also schedulable as a worker, which is the
k3s default) and two agents.

## Sizing

Defaults target a 16 GB host and leave headroom for Proxmox:

| node          | role          | vCPU | RAM   |
|---------------|---------------|------|-------|
| k3s-cp-01     | server+worker | 2    | 4 GB  |
| k3s-worker-01 | worker        | 2    | 4 GB  |
| k3s-worker-02 | worker        | 2    | 4 GB  |

The full data stack (Trino, Prometheus, Dagster, Garage) is memory hungry. On
16 GB it will be tight; raise `nodes` memory/cores in `terraform.tfvars` once the
host has more RAM, and attach `data_disk_gb` for Garage/Iceberg storage.

## Prerequisites

- Packer template built (VM ID `template_vm_id`, default 9000).
- The `snippet_datastore_id` allows the `snippets` content type. With
  `pve-bootstrap` this is enabled on `sda-data`.
- An API token for `terraform@pve` (role `TerraformProvisioner`).
- SSH key for the `ansible` user on the node (used only to upload snippets).
- The routed `vmbr10` network in place if using 10.x addresses.

## Usage

```sh
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars

terraform init
# Drive plan/apply through the root Makefile so the workload SSH CA public key
# is injected from ssh-ca/workload/ca.pub (the single source, no '..' read here):
make k3s-plan      # from the repo root
make k3s-apply
```

`ssh_ca_public_key` is a required variable with no default, so a bare
`terraform apply` is rejected. Always go through `make`.

Fetch the kubeconfig (command is also a Tofu output):

```sh
ssh deploy@10.0.0.10 sudo cat /etc/rancher/k3s/k3s.yaml | sed 's/127.0.0.1/10.0.0.10/' > k3s.yaml
export KUBECONFIG=$PWD/k3s.yaml
kubectl get nodes
```

## Notes

- k3s installs at first boot via cloud-init and needs internet on the node
  subnet (the install pulls from get.k3s.io). Agents retry until the server is
  up, so boot order does not matter.
- The join token is generated by `random_password` and kept in state. Keep the
  state file private.
- Manage the cluster only through this configuration; editing the VMs in the PVE
  GUI will drift from state.
